# Serve the petrodb landing at `sumpalabs.com/petrodb` via the sumpalabs umbrella router

**Status:** accepted — refines the landing-host consequence of [ADR-0003](0003-cloudflare-pages-hosting.md) and [ADR-0005](0005-host-parquet-on-huggingface.md)

## Context

ADR-0005 keeps the human-facing landing + schema docs on Cloudflare Pages (only the parquet *bytes* moved to Hugging Face). That landing was published at the subdomain `petrodb.ocortez.com`. petrodb is now one product under the **sumpalabs** umbrella (alongside future petroviz / formulaml / formuladb), and the company site lives at `sumpalabs.com`. To consolidate SEO authority into the apex domain, the landing must be served at the **subpath** `https://sumpalabs.com/petrodb/`, same-origin — not at a subdomain and not via a redirect, both of which dilute or forfeit the link-equity consolidation that is the entire reason for the move.

## Decision

Keep petrodb as its **own Cloudflare Pages project** (per ADR-0003), but strip its public custom domain. The slim Wrangler deploy (landing + per-dataset schema docs) stages its content **under a `/petrodb/` path** so the origin path equals the public path. A **Cloudflare Worker on the `sumpalabs.com` zone** — the *sumpalabs umbrella router* — owns the route `sumpalabs.com/petrodb*` and **reverse-proxies** it (same-origin, never a 30x) to the petrodb Pages deployment. Because the landing's internal links are all relative and origin path == public path, no link rewriting or `<base href>` is needed; only a trailing-slash canonical (`/petrodb/`) plus `<link rel="canonical">` / `og:url` pointing at `https://sumpalabs.com/petrodb/`.

The umbrella router is **shared apex infrastructure** and lives in the `sumpalabs-landing` repo, not here. It generalises to a prefix→origin map (`/petrodb*`, `/petroviz*`, …) so each future product can be added as one Pages project + one route entry. The canonical record of that umbrella pattern is an ADR in `sumpalabs-landing`; this ADR records only petrodb's participation in it. The HF data path (`BASE_URL_MAIN/STAGE` → HF `resolve`) is unaffected — only the landing host moves.

## Considered alternatives

- **`petrodb.sumpalabs.com` (subdomain).** Trivial — one custom domain on the Pages project, no Worker. Rejected: in practice subdomains are treated as a semi-separate site for authority, so it would not consolidate SEO into `sumpalabs.com` the way a subdirectory does, which is the whole motivation.
- **Redirect / Rewrite rule `sumpalabs.com/petrodb` → Pages domain.** Rejected: a redirect points the canonical content at the other host and forfeits the SEO benefit; rewrite rules can't pull a *different* Pages project's content and return it same-origin.
- **Merge the landing into the `sumpalabs-landing` Hugo project (monorepo / submodule).** No Worker needed, but petrodb's schema docs are generated by its own CI pipeline, so it would have to cross-publish built artifacts into the apex repo — coupling two independent deploys and dissolving petrodb's own Pages project. Rejected in favour of keeping the products decoupled behind a thin shared router.
- **Per-product Worker-with-static-assets (no Pages project).** Cloudflare's strategic direction, and low-cost for petrodb (it already has CI + wrangler). Rejected as the *umbrella standard*: it would force every simple marketing/docs site into Worker tooling and grant each product's CI route-management rights on the shared zone. The shared-router + per-product-Pages split keeps simple sites on frictionless Pages git-integration.

## Consequences

- The public landing URL in docs / the HF dataset card becomes `https://sumpalabs.com/petrodb/`; `petrodb.ocortez.com` is retired (301 → the new URL to preserve existing equity).
- petrodb's deploy and cutover now depend on the umbrella router existing on the `sumpalabs.com` zone — a cross-repo dependency tracked from petrodb's issue tracker.
- The petrodb Pages project is no longer publicly reachable on its own custom domain; it is exercised standalone only at its `*.pages.dev/petrodb/` URL for testing.
